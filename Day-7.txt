TCL(transaction control language)
----------------------------------
def
----
The process of combining all related operations into a single unit and executing them according to the rule "either all operations should be succeed or none should be succeed". is called Transaction management.

A transaction is considered a single unit of work and follows the principle

"either all or none"

example: Funds transfer.

Consider a money transfer process between two accounts.

1. Debit amount from the sender's account
2.Credit amount into the receiver's account.

Both operations should be executed as a single transaction.

If the amount is debited from the sender's account successfully, but crediting the recover's account fails. It may lead to the data inconsistency problems.

There are 2 types of transactions such as:

1.Local Transactions
-----------------------
In local transactions, all operations of a transaction are executed in the same database.

example: Funds transfer from one account to another account where both accounts belongs to the same bank is called local transactions.

2.Global transactions
------------------------
In global transactions, operations of a transaction are executed on different databases.
 
example: Funds transfer from one account to another account where both accounts belongs to the different banks is called global transactions. 


ACID Properties:#notes
-----------------------
1. Atomicity: All or nothing
2. Consistency: The database remains in a valid state
3. Isolation: Transactions do not interface with each other
4. Durability: Committed changes are permanent(time)

give lengthy explanation and real-world example  for theses properties

Atomicity: Operations within a transaction must all succeed or all fail. It acts as an "all or nothing" rule.
Real-World Example: Transferring ₹5,000 from Account A to Account B requires two steps (deducting from A, adding to B). 
If the server crashes after deducting from A but before adding to B, atomicity ensures the deduction is reversed so no money is lost.

Consistency: A transaction must strictly follow all predefined database rules and constraints. 
It ensures the database goes from one valid state to another.
Real-World Example: A bank's rule might state that no account can have a negative balance. 
Consistency ensures that any transaction violating this rule is blocked and canceled.

Isolation: Multiple concurrent transactions must execute independently without interfering with one another. 
It ensures processes run as if they have exclusive access to the system.
Real-World Example: Two customers buying the final ticket for the same movie at the exact same moment. 
Isolation ensures the system processes one after the other, preventing both users from being successfully booked into the same seat.

Durability: Once a transaction is successfully completed, its changes are permanently recorded in the system, 
even during a sudden power outage or server crash.
Real-World Example: After receiving an online shopping order confirmation, you turn off your computer. Because of durability, your order 
and payment data are safely stored in the database's non-volatile memory.
